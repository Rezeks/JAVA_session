## 📖 Сценарий демонстрации (7–10 минут)

### **Этап 1: Аутентификация и управление доступом (2 мин)**

**Страница:** `http://localhost:8080/ui/users.html`

**Шаг 1.1 — Регистрация двух пользователей**

1. Введи:
   - Login: `alice`
   - Password: `secure_password_123`
   - Нажми **"Регистрация"**

   Ожидаемый результат:
   ```
   ✅ User registered: alice
   Generated RSA KeyPair for alice
   Added to offline queue
   ```

2. Повтори для второго пользователя:
   - Login: `bob`
   - Password: `bob_secret_789`

**Шаг 1.2 — Подтверждение учётных записей**

1. Для каждого пользователя:
   - Login: `alice`
   - Нажми **"Подтвердить (confirm)"**

   Ожидаемый результат:
   ```
   ✅ User confirmed: alice
   Status: ACTIVE
   ```

**Шаг 1.3 — Аутентификация с 2FA**

1. Login: `alice`
2. Password: `secure_password_123`
3. Hardware Token: возьми из response после регистрации (или используй `00112233`)
4. Нажми **"Authenticate (2FA)"**

   Ожидаемый результат:
   ```
   ✅ AUTH_OK_2FA: alice has passed 2FA
   ```

**Что показывать:**

- ✅ Каждый пользователь получает свою пару RSA-2048 ключей (в KeyVault)
- ✅ Пароли хешируются PBKDF2-HMAC-SHA256 (100k итераций, 32-byte salt)
- ✅ 2FA требует аппаратный токен + пароль

### **Этап 2: Защита переписки (шифрование) (3 мин)**

**Страница:** `http://localhost:8080/ui/messages.html`

**Шаг 2.1 — Отправка защищённого сообщения**

1. Заполни:
   - Sender ID: скопируй `alice` ID из предыдущего этапа (или сгенерируй new)
   - Recipient ID: скопируй `bob` ID
   - Message: `"Это секретное сообщение, которое защищено AES-256-GCM + RSA"`

2. Нажми **"Отправить сообщение"**

   Ожидаемый результат:
   ```
   ✅ Message sent!
   Message ID: [UUID]
   Status: QUEUED
   
   Encrypted payload: [очень длинная строка base64]
   Wrapped message key: [очень длинная строка base64]
   Signature: [очень длинная строка base64]
   ```

**Что показывать:**

- ✅ **OneTimeKey**: генерируется случайно (256-bit) для каждого сообщения
- ✅ **Encrypted**: сообщение шифруется AES-256-GCM с этим ключом
- ✅ **WrappedKey**: OneTimeKey оборачивается в публичный ключ Боба (RSA-OAEP)
  - Только Боб может раскрыть этот ключ своим приватным ключом
- ✅ **Signature**: весь зашифрованный текст подписывается приватным ключом Алисы (SHA-256 with RSA)
  - Боб проверит, что это действительно от Алисы

**Шаг 2.2 — Подтверждение доставки**

1. Скопируй Message ID из результата выше
2. Заполни:
   - Message ID: [скопированный ID]
3. Нажми **"Подтвердить доставку (deliver)"**

   Ожидаемый результат:
   ```
   ✅ Message delivered!
   Status: DELIVERED
   ```

**Шаг 2.3 — Проверка истории и офлайн-пулла**

1. Заполни:
   - User ID: ID боба
2. Нажми **"История сообщений"**

   Ожидаемый результат:
   ```
   ✅ Messages for bob:
   - ID: [UUID]
     From: alice
     Status: DELIVERED
     Encrypted: [base64]
   ```

3. Нажми **"Пулл офлайн сообщений"**

   Ожидаемый результат:
   ```
   ✅ Offline messages for bob: 1
   - Message ID: [UUID]
   ```

**Что показывать:**

- ✅ Офлайн-очередь: сообщения хранятся на сервере, пока получатель оффлайн
- ✅ История: всех сообщений отслеживается (QUEUED → SENT → DELIVERED → READ)
- ✅ Шифрование: даже если сервер взломают, они видят только base64 гарнир

### **Этап 3: Групповые чаты (1.5 мин)**

**Страница:** `http://localhost:8080/ui/groups.html`

**Шаг 3.1 — Создание групповой беседы**

1. Заполни:
   - Owner ID: alice
   - Group name: `"Семейный чат"`
   - Members (JSON): `["alice_id", "bob_id", "charlie_id"]` (или просто имена UUID)

2. Нажми **"Создать группу"**

   Ожидаемый результат:
   ```
   ✅ Group created!
   Group ID: [UUID]
   Name: Семейный чат
   Members: 3
   ```

**Шаг 3.2 — Отправка группового сообщения**

1. Заполни:
   - Sender ID: alice
   - Group ID: [скопированный от создания]
   - Message: `"Привет, ребята! Это зашифровано для всей группы."`

2. Нажми **"Отправить в группу"**

   Ожидаемый результат:

**Что показывать:**

- ✅ Но OneTimeKey оборачивается отдельно для каждого члена группы (RSA-OAEP)

### **Этап 4: Передача файлов (2 мин)**

**Страница:** `http://localhost:8080/ui/files.html`

**Шаг 4.1 — Инициация загрузки**

1. Заполни:
   - Sender ID: alice
   - Recipient ID: bob
   - File name: `"secret_document.txt"`
   - File size: `1024` (bytes)

2. Нажми **"Инициировать загрузку"**

   Ожидаемый результат:
   ```
   ✅ Upload initiated!
   Transfer ID: [UUID]
   OneTimeKey: [base64]
   Max size: 2GB
   ```

**Шаг 4.2 — Загрузка чанка (chunk)**

1. Скопируй Transfer ID из выше
2. Заполни:
   - Transfer ID: [скопированный]
   - Offset: `0`
   - Chunk: `"Hello, this is secret content"` (или просто текст)

3. Нажми **"Загрузить чанк"**

   Ожидаемый результат:
   ```
   ✅ Chunk uploaded!
   Transfer ID: [UUID]
   Offset: 0
   Size uploaded: 29 bytes
   Checksum: [SHA-256 hash]
   ```

**Шаг 4.3 — Финализация и проверка**

1. Заполни:
   - Transfer ID: [скопированный]
   - Filename: `"secret_document.txt"`
   - Total size: `29`

2. Нажми **"Финализировать загрузку"**

   Ожидаемый результат:
   ```
   ✅ Upload finalized!
   File encrypted and stored
   Checksum: [SHA-256]
   ```

3. Проверка целостности:
   - Transfer ID: [скопированный]
   - Checksum: [скопируй из результата выше]
   - Нажми **"Проверить хеш"**

   Ожидаемый результат:
   ```
   ✅ Checksum verified!
   File integrity: OK
   ```

**Что показывать:**

- ✅ Файлы шифруются потоком (AES-GCM stream).
- ✅ Можно загружать по частям (chunks) с resume-поддержкой (offset).
- ✅ SHA-256 checksum позволяет проверить целостность на получающей стороне.
- ✅ Даже если файл 2 ГБ — он шифруется в потоке без загрузки в память целиком.

## 💬 Что давай сказать авдитории

### **При показе регистрации:**

> "Когда пользователь регистрируется, система генерирует ему пару RSA-2048 ключей. Приватный ключ остается в хранилище (KeyVault) и никуда не отправляется. Публичный ключ используется получателями для шифрования сообщений именно для этого пользователя."

### **При показе отправки сообщения:**

> "Жмём отправку: генерируется случайный 256-битный ключ, сообщение шифруется AES-256-GCM (это authenticated encryption — любые изменения будут обнаружены). Сам ключ оборачивается в публичный ключ получателя (RSA-OAEP) — только он сможет его раскрыть. И всё это подписывается приватным ключом отправителя (SHA-256 with RSA) — чтобы получатель знал, что сообщение точно от нас."

### **При показе истории:**

> "Даже все сообщения, которые мы видим на сервере — это шифротекст. Если сервер взломают (SQL injection, breach и т.д.), они видят только base64 гарнир. Расшифровать могут только отправитель и получатель, потому что приватные ключи у них."

### **При показе офлайн-пулла:**

> "Пользователь может быть оффлайн неделю — все сообщения, которые ему отправили, будут лежать в очереди на сервере. Когда он вернётся в сеть, просто натянет все сообщения и расшифрует."

### **При показе групповых чатов:**

> "В групповом чате одно сообщение шифруется один раз AES-GCM. Но OneTimeKey оборачивается отдельно для каждого члена группы в его публичный ключ. То есть одно физическое сообщение на сервере, но каждый может расшифровать только своим ключом."

### **При показе файлов:**

> "Файлы > 2 Гигабайт? Не проблема. Шифруются потоком — не загружаются в память целиком. Можно резюмировать загрузку: прошла ошибка на 50% — жмём Upload с тем же offset=50%, продолжаем. SHA-256 хеш позволяет получателю проверить, что файл пришёл целиком и не поменялся."

### **При показе аудита:**

> "Это журнал всех операций. В production такой журнал пишется в защищённый лог-стор (может быть даже в отдельной системе логирования). Для forensics: кто что и когда делал — полный трейл."

## 🎯 Чтобы быстро отвечать на вопросы

### **Q: Как это работает? Я не вижу никакого диалога как в Telegram.**

**A:** Это не полноценный мессенджер — это каркас (skeleton) защищённого мессенджера. Вся криптография сделана, архитектура правильная. UI просто показывает вызовы API. В production добавить WebSocket + real-time UI будет элементарно.

### **Q: Если я украду приватный ключ получателя, разшифруюправлю старые сообщения?**

**A:** Нет! Потому что каждое сообщение шифруется уникальным одноразовым ключом. Даже если украдёте приватный ключ сегодня, он откроет только новые сообщения (через RSA-OAEP unwrap). Старые сообщения были зашифрованы старыми одноразовыми ключами — они потеряны навсегда. Это Forward Secrecy.

### **Q: А если сервер взломают? Мои сообщения в опасности?**

**A:** Нет. На сервере хранится только шифротекст (base64 гарнир). Без приватных ключей отправителя/получателя расшифровать невозможно. Ключи хранятся отдельно и в production в HSM (аппаратный модуль защиты), который никогда не отправляет приватные ключи за пределы модуля.

### **Q: Как вы защищены от MITM (перехвата трафика)?**

**A:** Это уже в коде через сигнатуры (каждое сообщение подписано приватным ключом отправителя). В production добавляется mTLS (оба конца соединения проверяют друг друга через сертификаты).

### **Q: Почему когда я перезапущу сервер, все сообщения теряются?**

**A:** Потому что это demo версия. Всё в памяти JVM. В production добавится PostgreSQL и всё будет на диске.
