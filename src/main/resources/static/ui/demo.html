🏗️ Архитектура системы

Это защищённая система обмена сообщениями. Все данные шифруются E2E (end-to-end), пароли хешируются PBKDF2, доступ по JWT-токенам. База данных — PostgreSQL, очереди — Kafka.

🔑 Аутентификация и JWT

📝 Шаг 1: Регистрация

При регистрации пароль хешируется PBKDF2-SHA256 (100 000 итераций + соль). Это защищает даже при утечке БД — восстановить пароль невозможно.
1. Ввод данных
2. PBKDF2 хеш
3. Сохранение в БД
4. Генерация JWT

🔓 Шаг 2: Вход (Логин)

Логин проверяет: 1) существует ли пользователь, 2) совпадает ли хеш пароля, 3) верен ли 2FA токен. При 5 ошибках — автоблокировка.
1. Ввод credentials
2. Проверка PBKDF2
3. Проверка 2FA
4. Выдача JWT

🎫 Текущий JWT токен

JWT (JSON Web Token) — подписан HMAC-SHA256. Содержит login, role, время выпуска. Действует 1 час. Без него API возвращает 401.
Токен появится после регистрации или входа...

✉️ Защищённые сообщения

📤 Отправка сообщения

Для каждого сообщения генерируется уникальный AES-256-GCM ключ. Этот ключ оборачивается (wrap) публичным RSA-ключом получателя. Текст подписывается приватным RSA-ключом отправителя (SHA256withRSA).
1. AES-ключ
2. GCM encrypt
3. RSA wrap
4. RSA sign
5. Сохранение

📥 История и расшифровка

Получатель скачивает зашифрованные сообщения, разворачивает AES-ключ своим приватным RSA-ключом, расшифровывает текст AES-GCM, проверяет RSA-подпись отправителя.

🛡️ Сравнение методов шифрования

Тест шифрует и расшифровывает текст каждым из 7 алгоритмов, измеряет время и проверяет корректность. AES-256-GCM — стандарт проекта. DES показан для демонстрации уязвимости.

📊 Журнал аудита

Каждое действие записывается в таблицу audit_events в PostgreSQL: регистрация, аутентификация, отправка сообщений, ошибки. Это обеспечивает полную трассируемость и compliance.
